# Язык запросов DiGr

## Назначение языка

DSL в DiGr это прикладной язык запросов к уже построенному `AstDocument`.

Он не задаёт структуру документа заново. Структура документа определяется YAML-конфигом формата в `document_ast`, а DSL использует те сущности, которые реально присутствуют в дереве:

- `word`
- `symbol`
- `clause`
- `sentence`
- `paragraph`
- `page`
- `chapter`
- `slide`
- любые другие сущности, которые определены конфигом формата и реально попали в AST
- системная сущность `symbol`, которая автоматически создаётся под листовыми узлами AST

Из этого следуют два принципа:

1. язык не хардкодит конкретную модель документа;
2. корректность имён сущностей проверяется не только грамматикой, но и структурой реального `AstDocument`.

## Что уже реализовано

В проекте уже реализованы:

- лексический разбор текста DSL;
- синтаксический разбор в AST языка;
- выполнение запросов `FIND` и `CONTEXT` по `AstDocument`;
- валидация имён сущностей относительно документа;
- сериализуемые результаты выполнения.

То есть DSL в текущем проекте это уже не только спецификация, а рабочая подсистема.

## Две формы запроса

Сейчас язык поддерживает три верхнеуровневые формы:

- `FIND`
- `CONTEXT`
- `DISTANCE`

### `FIND`

`FIND` выбирает существующие узлы AST.

Пример:

```dsl
FIND sentence
WHERE has_descendant(word[text ~= /телефон/i])
RETURN text, count
```

Смысл:

- взять все узлы `sentence`;
- оставить только те, где в поддереве есть слово, совпадающее с `/телефон/i`;
- вернуть текст найденных предложений и общее количество совпадений.

### `CONTEXT`

`CONTEXT` ищет окна из соседних узлов одной базовой сущности.

Пример:

```dsl
CONTEXT sentence[<=4]
FOR "дверь", "тишина"
WITHIN paragraph[=1]
RETURN text, matches
```

Смысл:

- взять предложения как единицу окна;
- рассматривать окна длиной до четырёх предложений;
- потребовать совместное присутствие `дверь` и `тишина`;
- не выходить за пределы одного абзаца;
- вернуть текст окна и найденные совпадения.

### `DISTANCE`

`DISTANCE` измеряет расстояние между парами AST-узлов.

Пример:

```dsl
DISTANCE semantic_block[metadata.kind = "theorem"]
TO semantic_block[metadata.kind = "example"]
LIMIT_PAIRS all_nearest
RETURN pairs, stats, distance(semantic_block), count
```

Смысл:

- взять все блоки-теоремы и блоки-примеры;
- построить пары непересекающихся узлов;
- посчитать количество `semantic_block` строго между элементами пары;
- вернуть ближайшие пары и статистику по расстояниям.

## Общая модель исполнения

### Как исполняется `FIND`

Исполнение `FIND` идёт так:

1. выбираются все узлы указанной сущности;
2. для каждого узла вычисляется `WHERE`, если он указан;
3. все подходящие узлы попадают в результат;
4. `RETURN` влияет на форму результата, но не на сам факт совпадения.

### Как исполняется `CONTEXT`

Исполнение `CONTEXT` идёт так:

1. выбираются все узлы базовой сущности из `CONTEXT <entity>[...]`;
2. из них строятся непрерывные окна соседних узлов;
3. окно проверяется на ограничение длины;
4. окно проверяется на все паттерны из `FOR`;
5. окно проверяется на все ограничения `WITHIN`;
6. если задан `WHERE`, он вычисляется уже для найденного окна;
7. в итоговом результате сохраняются минимальные удовлетворяющие окна.

Последний пункт важен: executor специально убирает более широкие окна, если уже найдено вложенное более короткое окно с тем же смыслом.

## `FIND`: выборка узлов

### Общая форма

```dsl
FIND entity_name
WHERE predicate
WITHIN entity_name[count_constraint]
RETURN return_item, ...
```

В текущей реализации:

- `WHERE` опционален;
- `WITHIN` опционален;
- `RETURN` опционален;
- фильтрация всегда идёт по узлам одной указанной сущности.

### Пример 1

```dsl
FIND sentence
WHERE follows(sentence[text ~= /Только тишина/])
RETURN text
```

Текстовая интерпретация:

- пройти по всем предложениям;
- для каждого предложения посмотреть, следует ли оно непосредственно за предложением, где встречается `Только тишина`;
- вернуть текст найденного предложения.

### Пример 2

```dsl
FIND paragraph
WHERE text ~= /пообещал/
  AND text ~= /фотографи.*/
RETURN text, span
```

Текстовая интерпретация:

- пройти по всем абзацам;
- оставить только те, где одновременно упоминаются обещание и фотография;
- вернуть текст абзаца и его координаты.

## `CONTEXT`: поиск контекстных окон

### Общая форма

```dsl
CONTEXT entity_name[count_constraint]
FOR pattern, pattern, ...
WITHIN entity_name[count_constraint]
WHERE predicate
RETURN return_item, ...
```

### Что такое окно

Контекстное окно это:

- последовательность подряд идущих узлов одного типа;
- без пропусков;
- в порядке следования в документе.

Например:

```dsl
CONTEXT sentence[<=3]
```

означает, что рассматриваются окна длины 1, 2 или 3 предложения.

Если в документе есть `S1, S2, S3, S4`, допустимы окна:

- `S1`
- `S1..S2`
- `S1..S3`
- `S2`
- `S2..S3`
- `S2..S4`

Но недопустимы:

- `S1 + S3`
- `S1 + S4`

### Пример

```dsl
CONTEXT sentence[<=4]
FOR "дверь", "тишина"
WITHIN paragraph[=1]
RETURN text, matches
```

Текстовая интерпретация:

- искать короткие окна из подряд идущих предложений;
- в каждом окне должны встретиться оба паттерна;
- окно не должно выходить за пределы одного абзаца;
- из всех подходящих окон вернуть минимальные.

## Паттерны в `FOR`

В `FOR` сейчас поддерживаются три формы паттернов.

### Строковый литерал

```dsl
FOR "ключ"
```

Смысл:

- в тексте окна должна встретиться строка `"ключ"`.

### Regex

```dsl
FOR /ассоциативн.*/
```

Смысл:

- в тексте окна должно встретиться совпадение регулярного выражения.

### Селектор

```dsl
FOR word[text = "ключ"]
```

Смысл:

- внутри окна должен существовать хотя бы один узел `word`, удовлетворяющий предикату.

### Алиасы

Паттерн может иметь alias:

```dsl
FOR a: word[text = "ключ"], b: word[text ~= /память/i]
```

Alias сохраняется в AST запроса и затем попадает в `matches` результата `CONTEXT`, поэтому его удобно использовать в приложениях поверх DSL.

## `WITHIN`: структурные ограничения

`WITHIN` ограничивает окно или найденный узел по контейнерам.

Пример:

```dsl
WITHIN paragraph[=1]
WITHIN page[=1]
```

Смысл:

- результат должен лежать внутри одного абзаца;
- и одновременно внутри одной страницы.

### Формальная семантика

Для каждого ограничения берутся все контейнеры указанного типа, являющиеся предками рассматриваемых узлов. Затем считается число различных контейнеров этого типа.

Примеры:

- `paragraph[=1]` означает ровно один абзац;
- `paragraph[<=2]` означает не более двух абзацев;
- `page[>1]` означает, что результат охватывает больше одной страницы.

## `WHERE`: логические условия

`WHERE` используется:

- в `FIND` как основной фильтр;
- в `CONTEXT` как дополнительный фильтр уже найденного окна;
- внутри селекторов.

Поддерживаются:

- `NOT`
- `AND`
- `OR`

Приоритет:

1. скобки;
2. `NOT`;
3. `AND`;
4. `OR`.

Пример:

```dsl
WHERE has_descendant(word[text = "телефон"])
  AND NOT has_descendant(word[text = "тишина"])
```

## Селекторы

Селектор это ссылка на AST-узлы по имени сущности с опциональным предикатом.

Формы:

```dsl
word
word[text = "ключ"]
sentence[has_descendant(word[text = "ключ"])]
```

Селектор без предиката означает все узлы указанной сущности в текущей области поиска.

Селектор с предикатом означает все узлы этой сущности, удовлетворяющие логическому условию.

## Поля и сравнения

В текущей реализации DSL поддерживаются поля:

- `entity`
- `text`
- `start`
- `end`
- `metadata.<field>`

Поддерживаемые операторы:

- `=`
- `!=`
- `<`
- `<=`
- `>`
- `>=`
- `~=`
- `!~=`

Примеры:

```dsl
text = "ключ"
text ~= /память/i
start >= 100
metadata.kind = "quote"
```

## Встроенные функции

Сейчас executor поддерживает следующие функции:

- `has_child(selector)`
- `has_descendant(selector)`
- `has_ancestor(selector)`
- `contains(selector)`
- `follows(selector)`
- `precedes(selector)`
- `intersects(selector)`
- `inside(selector)`

### Пример `has_descendant`

```dsl
FIND sentence
WHERE has_descendant(word[text ~= /телефон/i])
RETURN text
```

Смысл:

- взять предложение;
- проверить, есть ли внутри его поддерева подходящий узел `word`.

### Пример `follows`

```dsl
FIND sentence
WHERE follows(sentence[text ~= /Только тишина/])
RETURN text
```

Смысл:

- выбрать предложение, которое идёт непосредственно после предложения, удовлетворяющего селектору.

## Практическая оговорка про текущий `txt.yaml`

Для текущего `config/formats/txt.yaml` структура такая:

```text
document -> page -> paragraph -> sentence -> clause -> word
```

Из этого следует важная практическая деталь:

```dsl
FIND sentence
WHERE has_descendant(word[text = "телефон"])
```

для `txt` обычно корректно, а запрос

```dsl
FIND sentence
WHERE has_child(word[text = "телефон"])
```

чаще всего не сработает, потому что прямые дети `sentence` это `clause`, а не `word`.

Это не ограничение языка, а следствие конфигурации конкретного формата.

## `RETURN` и форма результата

`RETURN` управляет тем, что попадёт в сериализуемый результат.

Практически полезные элементы:

- `text`
- `span`
- `matches`
- `count`
- `window`

В текущей реализации наружу возвращаются:

- `FindQueryExecutionResult` для `FIND`;
- `ContextQueryExecutionResult` для `CONTEXT`.

Оба результата поддерживают `to_dict()`.

## Встроенные функции

Текущий executor поддерживает следующие функции.

### `has_child(selector)`

Проверяет наличие непосредственного ребёнка, подходящего под selector.

```dsl
has_child(clause[text ~= /тишина/])
```

### `has_descendant(selector)`

Проверяет наличие потомка на любой глубине.

```dsl
has_descendant(word[text ~= /телефон/i])
```

### `has_ancestor(selector)`

Проверяет наличие предка.

```dsl
has_ancestor(paragraph[text ~= /обещал/])
```

### `contains(argument)`

Проверяет наличие содержимого в пределах текущего узла.

Аргументом может быть:

- строка;
- regex;
- selector;
- field reference.

Примеры:

```dsl
contains("тишина")
contains(/телефон/i)
contains(word[text = "ключ"])
```

### `follows(selector)`

Проверяет, что предыдущий соседний узел того же типа соответствует selector.

```dsl
follows(sentence[text ~= /Только тишина/])
```

### `precedes(selector)`

Проверяет, что следующий соседний узел того же типа соответствует selector.

### `intersects(selector)`

Проверяет пересечение по span.

### `inside(selector)`

Проверяет, что текущий узел лежит внутри контейнера, соответствующего selector.

## `RETURN`: что вернуть

`RETURN` определяет форму результата.

Поддержанные элементы:

- `window`
- `text`
- `span`
- `matches`
- `nodes`
- `count`
- `source`
- `pairs`
- `stats`
- `distance(entity)`

### Для `FIND`

Обычно используются:

- `text`
- `nodes`
- `span`
- `count`

### Для `CONTEXT`

Обычно используются:

- `window`
- `text`
- `span`
- `matches`
- `distance(entity)`
- `count`

Если в `CONTEXT` указан `RETURN distance(entity)`, executor добавляет поле `distances` для selector-паттернов из `FOR`.
Строковые и regex-паттерны не участвуют в расчёте расстояний, потому что у них нет AST-узла.

### Для `DISTANCE`

Обычно используются:

- `pairs`
- `stats`
- `distance(entity)`
- `count`

`distance(entity)` обязателен для `DISTANCE`: он задаёт, узлы какого типа считать единицей расстояния.
Например, `distance(word)` считает слова между парой, `distance(sentence)` считает предложения, а `distance(symbol)` считает символы.

`LIMIT_PAIRS` управляет объёмом результата:

- отсутствует или `nearest` — одна ближайшая пара;
- `all_nearest` — все пары с минимальным расстоянием;
- целое число — k ближайших пар;
- `all` — все валидные пары.

## Что важно учитывать на практике

DSL исполняется по реальному AST, а не по идеализированной модели.

Для текущего `txt.yaml` структура такая:

- `sentence`
  - `clause`
    - `word`

Из-за этого запрос:

```dsl
FIND sentence
WHERE has_child(word[text = "телефон"])
```

для `txt` не отражает реальную структуру дерева.

Практически корректный вариант:

```dsl
FIND sentence
WHERE has_descendant(word[text ~= /телефон/i])
RETURN text, count
```

Это важная оговорка: DSL строг относительно структуры AST.
