# Язык Запросов DiGr

## Назначение

Язык запросов DiGr это DSL поверх AST документа. Его задача не в том, чтобы заново описывать структуру документа, а в том, чтобы делать запросы к уже построенному дереву.

Структура документа задаётся отдельно через YAML-конфиг формата, а DSL работает с именами сущностей из этого конфига:

- `word`
- `sentence`
- `paragraph`
- `page`
- `slide`
- `chapter`
- и любые другие entity, определённые моделью документа

Это означает:

- DSL не хардкодит конкретную структуру текста
- один и тот же язык может работать для `txt`, `tex`, `xml`, `json`, презентаций и других форматов
- валидность имён сущностей определяется не грамматикой, а AST и `ParserConfig`

Формальная грамматика находится в [dsl_grammar.ebnf](/home/forthey/projects/DiGr/docs/dsl_grammar.ebnf). Этот документ описывает язык на уровне смысла и практического использования.

## Две формы запроса

В текущей спецификации есть две основные формы:

- `CONTEXT` для поиска контекстных окон
- `FIND` для прямой выборки узлов AST

## 1. `CONTEXT`

### Идея

`CONTEXT` нужен для основной задачи проекта: найти такой непрерывный фрагмент документа, в котором совместно присутствуют нужные понятия, и при этом выполняются структурные ограничения.

Общая форма:

```dsl
CONTEXT sentence[<=10]
FOR "ключ", "ассоциативная память"
WITHIN paragraph[=1]
RETURN window, matches
```

### Семантика

Запрос читается так:

- взять базовую сущность окна: `sentence`
- рассматривать непрерывные окна длиной не более `10` экземпляров этой сущности
- найти окна, где встречаются оба искомых паттерна
- оставить только те окна, которые укладываются в один `paragraph`

### Что такое окно

Окно в `CONTEXT` это:

- непрерывная последовательность узлов одного типа
- в порядке их следования в документе
- без пропусков внутри окна

Например, `sentence[<=10]` означает:

- окно состоит из подряд идущих `sentence`
- длина окна от `1` до `10`

Если в AST есть предложения `S1, S2, S3, S4`, то допустимыми окнами будут:

- `S1`
- `S1..S2`
- `S1..S3`
- `S2..S4`

Но не:

- `S1 + S3`
- `S1 + S4`

потому что это уже не непрерывное окно.

### Что значит `sentence[<=10]`

Часть:

```dsl
CONTEXT sentence[<=10]
```

означает, что единицей длины окна являются узлы `sentence`, а ограничение задаётся по их количеству.

Это не:

- не число символов
- не число слов
- не число токенов LLM

а именно количество AST-узлов данного типа.

### Как вычисляется текст окна

Текст окна определяется как минимальный непрерывный текстовый интервал, покрывающий все узлы окна:

- `start = min(start_i)`
- `end = max(end_i)`

для всех базовых узлов внутри окна.

## 2. `FIND`

### Идея

`FIND` это более общий запрос на выборку узлов определённого типа по предикату.

Пример:

```dsl
FIND sentence
WHERE has_child(word[text ~= /не.*/])
RETURN nodes, count
```

Смысл:

- пройти по всем `sentence`
- оставить только те, у которых есть дочерний `word`, начинающийся на `не`
- вернуть найденные узлы и их количество

### Когда использовать `FIND`, а когда `CONTEXT`

Используй `FIND`, если:

- нужен отбор узлов по условиям
- нет идеи контекстного окна
- важен сам набор найденных сущностей

Используй `CONTEXT`, если:

- нужна совместная встречаемость понятий
- важна дистанция между ними
- есть ограничения по абзацам, страницам, главам и т.д.

## `FOR`: что именно нужно найти

В `CONTEXT` секция `FOR` задаёт паттерны, которые должны встретиться в окне.

Простейший случай:

```dsl
FOR "ключ", "ассоциативная память"
```

Это sugar для текстового поиска по окну.

Поддерживаются три формы паттерна.

### 1. Строковый литерал

```dsl
FOR "ключ"
```

Смысл:

- в окне должен встретиться текст `"ключ"`

В базовой модели такой паттерн интерпретируется как текстовое совпадение по узлам, пригодным для текстовой проверки.

### 2. Regex

```dsl
FOR /ассоциативн.*/
```

Смысл:

- в окне должно встретиться совпадение регулярного выражения

### 3. Селектор

```dsl
FOR word[text = "ключ"]
```

Смысл:

- в окне должен существовать хотя бы один узел `word`, удовлетворяющий предикату `text = "ключ"`

Это наиболее точная форма, потому что она опирается не просто на строку окна, а на AST.

### Алиасы

Паттерн может иметь alias:

```dsl
FOR a: word[text = "ключ"], b: clause[text ~= /ассоциативная память/i]
```

Сейчас alias фиксируется как часть синтаксиса и нужен для будущих расширений языка:

- адресация совпадений в `RETURN`
- постобработка результатов
- выражение отношений между найденными объектами

В текущей v1-спецификации alias считается допустимым синтаксисом, но не является обязательной частью минимального executor.

## `WITHIN`: структурные ограничения

`WITHIN` задаёт ограничение на контейнеры, внутри которых должен лежать результат.

Пример:

```dsl
WITHIN paragraph[=1]
```

Смысл:

- все базовые узлы окна должны принадлежать одному и тому же `paragraph`

Пример:

```dsl
WITHIN paragraph[<=2]
WITHIN page[=1]
```

Смысл:

- окно может занимать не более двух абзацев
- но при этом должно лежать в пределах одной страницы

### Формальная семантика `WITHIN`

Для заданного запроса:

```dsl
CONTEXT sentence[<=10]
WITHIN paragraph[<=2]
```

берутся все `paragraph`, являющиеся ancestor-контейнерами узлов окна. Затем считается число различных контейнеров этого типа.

Ограничение:

- `=1` означает ровно один контейнер
- `<=2` означает не более двух различных контейнеров
- `>1` означает более одного контейнера

То есть `WITHIN paragraph[=1]` это ограничение не на текст как строку, а на структуру AST.

## `WHERE`: логические условия

`WHERE` используется:

- в `FIND` как главный фильтр
- в `CONTEXT` как дополнительный фильтр к уже найденному окну
- внутри селектора `entity[...]` как фильтр для конкретного узла

Пример:

```dsl
FIND sentence
WHERE has_child(word[text = "ключ"]) AND NOT has_child(word[text = "замок"])
```

### Приоритет операторов

Логический приоритет такой:

1. скобки
2. `NOT`
3. `AND`
4. `OR`

Пример:

```dsl
WHERE A OR B AND C
```

интерпретируется как:

```dsl
WHERE A OR (B AND C)
```

## Селекторы

Селектор это ссылка на AST-узлы по имени сущности с опциональным предикатом.

Формы:

```dsl
word
word[text = "ключ"]
sentence[has_child(word[text = "ключ"])]
```

### Семантика селектора без предиката

```dsl
word
```

означает множество всех узлов `word` в текущей области вычисления.

### Семантика селектора с предикатом

```dsl
word[text = "ключ"]
```

означает множество всех узлов `word`, у которых поле `text` равно `"ключ"`.

## Поля и сравнения

В текущем языке поле задаётся через `field_ref`.

Примеры:

```dsl
text = "ключ"
start >= 100
metadata.format = "txt"
```

Поддержанные операторы сравнения:

- `=`
- `!=`
- `~=`
- `!~=`
- `<`
- `<=`
- `>`
- `>=`

### Семантика `~=`

```dsl
text ~= /ключ.*/
```

Это regex-match.

### Семантика `!~=`

```dsl
text !~= /черновик/
```

Это отрицание regex-match.

## Встроенные функции

Текущая грамматика резервирует набор стандартных навигационных функций:

- `has_child`
- `has_descendant`
- `has_ancestor`
- `contains`
- `follows`
- `precedes`
- `intersects`
- `inside`

Ниже приведена рекомендуемая семантика v1.

### `has_child(selector)`

```dsl
has_child(word[text = "ключ"])
```

Истинно, если у текущего узла есть непосредственный дочерний узел, удовлетворяющий селектору.

### `has_descendant(selector)`

```dsl
has_descendant(word[text = "ключ"])
```

Истинно, если среди любых потомков текущего узла есть совпадение.

### `has_ancestor(selector)`

```dsl
has_ancestor(paragraph)
```

Истинно, если у текущего узла есть ancestor, удовлетворяющий селектору.

### `contains(selector)`

```dsl
contains(word[text = "ключ"])
```

Семантически близко к `has_descendant`, но более естественно читается для диапазонных сущностей вроде `sentence`, `paragraph`, `page`.

Для v1 допустимо считать `contains(selector)` эквивалентным `has_descendant(selector) OR selector matches self-contained text space`, если executor работает не только по узлам, но и по текстовому покрытию.

### `follows(selector)`

```dsl
FIND sentence
WHERE follows(sentence[contains(word[text = "тишина"])])
```

Истинно, если текущий узел следует после хотя бы одного узла, удовлетворяющего селектору.

Рекомендуемая семантика v1:

- сравнение идёт на одном уровне сущностей
- если текущий запрос это `FIND sentence`, то `follows(sentence[...])` сравнивает порядок предложений
- отношение определяется по порядку обхода документа

### `precedes(selector)`

Зеркально к `follows`.

### `intersects(selector)`

Истинно, если текстовые диапазоны `start:end` пересекаются.

### `inside(selector)`

Истинно, если текущий узел целиком лежит внутри диапазона узла, удовлетворяющего селектору.

## `RETURN`: форма результата

`RETURN` задаёт, что именно должен вернуть executor.

Примеры:

```dsl
RETURN window, matches
RETURN nodes, count
RETURN text, span, source
```

Рекомендуемая интерпретация элементов:

- `window`: структурное представление найденного окна для `CONTEXT`
- `text`: текст результата
- `span`: абсолютные позиции `start:end`
- `matches`: найденные паттерны и их локализация
- `nodes`: AST-узлы результата
- `count`: количество результатов
- `source`: информация об исходном документе

Если `RETURN` не указан, для v1 разумно считать значением по умолчанию:

- для `CONTEXT`: `window, matches`
- для `FIND`: `nodes`

## Нормативные примеры

### Пример 1. Два понятия в пределах 10 предложений

```dsl
CONTEXT sentence[<=10]
FOR "ключ", "ассоциативная память"
RETURN window, matches
```

### Пример 2. В пределах 10 предложений, но в одном абзаце

```dsl
CONTEXT sentence[<=10]
FOR "ключ", "ассоциативная память"
WITHIN paragraph[=1]
RETURN window, matches
```

### Пример 3. В пределах 10 предложений, максимум в двух абзацах и на одной странице

```dsl
CONTEXT sentence[<=10]
FOR "ключ", "ассоциативная память"
WITHIN paragraph[<=2]
WITHIN page[=1]
RETURN window, matches
```

### Пример 4. Для TeX-документа, не выходя за главу

```dsl
CONTEXT sentence[<=12]
FOR "ключ", "ассоциативная память"
WITHIN chapter[=1]
RETURN text, span
```

### Пример 5. Для презентации в пределах одного слайда

```dsl
CONTEXT bullet[<=8]
FOR "ассоциативная память", "ключ"
WITHIN slide[=1]
RETURN window
```

### Пример 6. Прямая выборка предложений

```dsl
FIND sentence
WHERE has_child(word[text ~= /не.*/])
RETURN nodes, count
```

### Пример 7. Выборка абзацев, содержащих оба понятия

```dsl
FIND paragraph
WHERE has_descendant(word[text = "ключ"])
  AND text ~= /ассоциативная память/i
RETURN text, span
```

## Рекомендуемая операционная семантика `CONTEXT`

Для `CONTEXT` v1 рекомендован такой алгоритм:

1. определить все базовые узлы указанной сущности, например `sentence`
2. построить все непрерывные окна, удовлетворяющие `span_spec`
3. отфильтровать окна по `WITHIN`
4. проверить выполнение всех паттернов `FOR`
5. применить `WHERE`, если он есть
6. вернуть результат в форме, заданной через `RETURN`

### Минимальные окна

Чтобы избежать большого числа дубликатов, для v1 разумно использовать правило минимальности:

- если окно `W1` удовлетворяет запросу
- и существует строго меньшее окно `W2`, покрывающее те же совпадения
- то возвращается `W2`, а `W1` отбрасывается

Это особенно важно для запросов вида:

```dsl
CONTEXT sentence[<=10]
FOR "ключ", "ассоциативная память"
```

иначе результат будет быстро разрастаться до большого количества вложенных окон.

## Что не входит в v1

В текущую спецификацию сознательно не включены:

- вычисление эмбеддингов
- классификация семантических отношений
- UML-отношения как часть ядра языка
- арифметика score и ranking
- пользовательские макросы
- агрегации вроде `GROUP BY`

Это можно добавлять поверх текущего ядра, но для первой версии языка они только усложнят реализацию и расплывут семантику.

## Граница между грамматикой и семантикой

Важно различать два уровня:

- [dsl_grammar.ebnf](/home/forthey/projects/DiGr/docs/dsl_grammar.ebnf) отвечает за допустимую форму записи
- этот документ отвечает за смысл конструкций

Например:

- грамматика разрешает `WITHIN chapter[=1]`
- а семантика требует, чтобы `chapter` реально существовал как entity в AST текущего документа

То же самое касается:

- имён полей
- пользовательских функций
- допустимых форм `RETURN`
